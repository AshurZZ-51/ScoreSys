# ScoreSys 立项评审系统 PRD 与交接说明

**文档版本：** 2026-07-24

**适用版本：** 项目池 V2 + 两轮评分 V3
**代码分支：** `codex/project-pool-v2`

## 1. 产品目标

ScoreSys 是用于游戏项目立项前评审的内部系统。系统以项目为主档，将资料检查、评审会安排、两轮评分、问题与行动、Walker 结论、项目状态、报告快照和历史归档串联为可追溯流程。

核心原则：

- `project_pool` 是真实项目的唯一主档；一场评审会只是承载该项目某一轮、某一次评审的场次。
- 同一项目通常至少经历两场评审会：第一轮通过后，由管理员手动加入新的评审会参加第二轮。
- 第一轮和第二轮各自独立计算，基础总分均为 100 分；Walker 加分单列，不改变基础分满分。
- 所有已加入该场评审会的评委均可在截止日期前持续评分和修改；Walker 结论不会隐藏其他评委尚未完成的评分项目。
- 一次评审仅在 Walker 保存“通过 / 重评 / 驳回”后，才计入项目评审历史、已评审项目池和结果池。
- 资料状态当前仅用于展示和管理，不阻止排入评审会。
- 历史记录只读兼容，不能用当前规则反算或篡改历史得分。

## 2. 角色与权限

| 角色 | 权限 |
| --- | --- |
| 超级管理员 `admin51` | 后台全部权限；账号管理；创建账号、重置密码、设置管理员；发起项目完全删除待清除。 |
| 普通管理员 | 项目池、资料检查、项目状态、评审会、报告、归档与恢复；无账号管理权限。 |
| Walker | 参与全部评分；填写问题与改进动作；填写加分及原因；保存当轮结论。 |
| 普通评委 | 对被安排项目填写当轮全部评分、问题与改进动作。 |

所有写接口必须在服务端校验登录身份、会议评委快照和角色权限，前端隐藏按钮不是权限边界。

## 3. 项目与评审会流程

### 3.1 待评审项目池

管理员先创建项目主档，维护项目名称、提报人、描述、资料状态、项目阶段状态和历史记录。待评审池显示所有尚需要评审的项目，并支持按月份筛选、批量勾选、批量改状态、批量加入评审会和批量归档。

项目可用状态：

| 状态 | 含义 | 可安排轮次 |
| --- | --- | --- |
| `draft` | 草稿 | 第一轮，第 1 次 |
| `materials_pending` | 资料待补充 | 第一轮，第 1 次 |
| `ready_r1` | 第一轮待安排 | 第一轮，第 1 次 |
| `r1_recheck_ready` | 第一轮重评待安排 | 第一轮，第 2 次 |
| `ready_r2` | 第二轮待安排 | 第二轮，第 1 次 |
| `r2_recheck_ready` | 第二轮重评待安排 | 第二轮，第 2 次 |
| `initiation` | 已通过两轮，进入立项流程 | 不再安排评审 |
| `rejected` | 已驳回 | 不再安排评审 |

管理员可人工调整项目状态，调整后应保留操作时间、操作者和可选备注。列表与详情页必须即时显示服务端返回的真实状态，不允许重新打开后回退为默认值。

### 3.2 资料检查

每个项目具有以下资料项：

| 资料项 | 必填 |
| --- | --- |
| 项目基础信息 | 是 |
| 项目定位 | 是 |
| 产品/玩法方案 | 是 |
| 风险说明 | 是 |
| 项目预算与排期计划 | 是 |
| 商业模式规划 | 否 |
| 资源需求 | 否 |
| 竞品或参考 | 否 |

每项资料只有四种状态：`missing`（缺失）、`needs_completion`（待完善）、`submitted`（已提交）、`exempt`（豁免）。全部必填项为“已提交”或“豁免”时，项目显示为“资料齐全”。

资料齐全状态必须在后台列表、项目详情和评委评分页面顶部展示；评委页展示完整资料表。当前不以此限制入会，未来若启用限制，应使用独立开关且不改写已有资料数据。

### 3.3 评审会管理

评审会不预设轮次，**项目在评审会中的 assignment 决定轮次**。创建或编辑评审会时，管理员从待评审项目池选择符合当前状态的项目，也可以快速新建项目后直接加入。

- 单场评审会最多 12 个有效项目，以 12 个槽位呈现。
- 项目编排支持拖拽排序并持久化。
- 当前评审会置顶并有明显标记，其他会议按日期从新到旧排序。
- 管理员可设置会议名称、会议日期、截止日期和备注；日期字段仅保存日期。
- 评审会列表支持勾选、批量移入回收站；回收站支持恢复。
- 有评分的项目 assignment 不应被直接移出，以保留审计完整性。

评审会详情工作区至少包含：会议设置、项目编排、实时结论与汇总、打印报告入口。评审会的实时汇总以服务端评分为准，不读取旧前端缓存。

## 4. 评分规则

### 4.1 通用规则

- 除创新性外，所有分项均为 0-10 分，允许录入小数。
- 一个普通维度的结果 = 所有评委在每个分项上的平均分，再按该维度系数换算。
- 创新性由每位评委选择档位，结果取所有有效档位的中位数；偶数人数时先求中位数，再取最接近合法档位，等距取较高档位。
- 轮次基础分由各维度结果相加，基础分最高 100 分。
- Walker 加分独立保存并附原因，显示为“基础分 + 加分 = 总分”；加分不改变“满分 100”的基础评分口径。
- 完成度按该项目所属评审会的评委快照和当轮应填分项计算。所有评委均参与所有当轮维度。

### 4.2 第一轮：游戏性 / 创新性

第一轮的新 assignment 使用 `two_round_v2` 规则，基础分 100 分。

| 维度 | 满分 | 分项与计算 |
| --- | ---: | --- |
| 游戏性 | 60 | 核心玩法、体感适配性、可复玩性、年龄适配度，各 0-10；四项平均分 × 6。 |
| 创新性 | 40 | L1-L5 档位，取评委档位中位数。 |

第一轮创新性档位：

| 档位 | 分值 | 含义 |
| --- | ---: | --- |
| L1 | 10 | 平台已有（完全复制） |
| L2 | 16 | 已有体感标杆 |
| L3 | 24 | 类似标杆 |
| L4 | 30 | 部分原创 |
| L5 | 40 | 完全原创（全自研独创） |

```text
第一轮基础分 = 游戏性得分 + 创新性得分
```

### 4.3 第二轮：五维综合复评

第二轮的新 assignment 使用 `two_round_v3` 规则。第二轮需要**再次评估全部五个维度**，基础分 100 分。

| 维度 | 满分 | 计算方式 |
| --- | ---: | --- |
| 游戏性 | 30 | 核心玩法、体感适配性、可复玩性、年龄适配度，各 0-10；四项平均分 × 3。 |
| 创新性 | 20 | L1-L5 档位，取评委档位中位数。 |
| 项目规划 | 20 | 项目里程碑、文档完善程度、风险难点分析、项目预算合理性、外包方案合理度，各 0-10；五项平均分 × 2。 |
| 技术&美术 | 15 | 技术方案评审、美术方案评审，各 0-10；两项平均分 × 1.5。 |
| 风险预估 | 15 | 延期风险、方向风险、资源风险、政策版权风险，各 0-10；四项平均分 × 1.5。 |

第二轮创新性档位：

| 档位 | 分值 | 含义 |
| --- | ---: | --- |
| L1 | 8 | 平台已有（完全复制） |
| L2 | 10 | 已有体感标杆 |
| L3 | 12 | 类似标杆 |
| L4 | 14 | 部分原创 |
| L5 | 20 | 完全原创（全自研独创） |

```text
第二轮基础分 = 游戏性 + 创新性 + 项目规划 + 技术&美术 + 风险预估
```

### 4.4 结论与重评

Walker 在每轮末给出 `approved`（通过）、`recheck`（重评）或 `rejected`（驳回）。每轮最多一次重评机会，第二次同轮评审不能再选择重评。

| 当前轮次 | 尝试 | 结论 | 结果状态 |
| --- | --- | --- | --- |
| 第一轮 | 第 1 次 | 通过 | `ready_r2` |
| 第一轮 | 第 1 次 | 重评 | `r1_recheck_ready` |
| 第一轮 | 第 1 次 | 驳回 | `rejected` |
| 第一轮 | 第 2 次 | 通过 | `ready_r2` |
| 第一轮 | 第 2 次 | 驳回 | `rejected` |
| 第二轮 | 第 1 次 | 通过 | `initiation` |
| 第二轮 | 第 1 次 | 重评 | `r2_recheck_ready` |
| 第二轮 | 第 1 次 | 驳回 | `rejected` |
| 第二轮 | 第 2 次 | 通过 | `initiation` |
| 第二轮 | 第 2 次 | 驳回 | `rejected` |

## 5. 评委评分页面

- 登录后仅显示当前评审会中分配给该评委的项目。
- 每个项目置顶展示资料检查表和资料齐全统计。
- 第一轮仅显示游戏性、创新性；第二轮显示完整五维复评。
- 所有评分、档位选择、问题、改进动作、Walker 加分和 Walker 结论都必须可保存，成功或失败应给予可见反馈。
- 数字输入允许清空和重新输入任意合法数值，失焦或明确提交时保存，不得回填错误默认分数。
- 问题与改进动作对所有评委开放，保存后实时汇总到后台。
- Walker 的加分和结论仅向 Walker 展示；结论位于 Walker 页面的项目底部。

## 6. 后台、结果与报告

后台的主板块：待评审项目池、评审会管理、结论与报告、已评审项目池、结果池、归档与恢复、账号管理。

- 已评审项目池仅显示至少有一条 Walker 结论、且已有对应分数的项目。
- 结果池按“通过 / 待重评 / 驳回”拆分为可展开列表；未评审项目仍留在待评审项目池。
- 结论与报告页使用评审会下拉选择器，不重复展示评审会管理列表；选择后显示该会实时的排名、维度对比、完成度、问题、行动和结论。
- 报告生成时保存快照，报告内容依据生成时的后台实时数据；允许后续重新生成新的快照版本。
- 第一轮报告展示第一轮维度、排名、完成度、问题行动和去向；第二轮 V3 报告展示完整五维、排名、完成度、问题行动和最终去向。
- 对 `initiation` 项目可以生成项目报告，包含两轮评审历史、会议、分数、排名、维度表现、问题行动、Walker 结论和状态时间线。

## 7. 归档、恢复与删除

- 归档只是把项目移出活跃池，不删除评分、意见、会议记录或报告。
- 归档、恢复、完全删除均需二次确认，并支持批量处理。
- 仅超管可以发起完全删除；项目先进入 15 天待清除区，可在期限内恢复。
- 到期后由受控清理任务物理删除项目、资料、状态历史、关联评审记录、评分、问题行动、报告快照与删除审计记录；到期后不可恢复。
- 评审会删除采用回收站逻辑，不进行自动物理删除，避免破坏历史评审链路。

## 8. 历史兼容与数据边界

| 评分版本 | 适用范围 | 兼容策略 |
| --- | --- | --- |
| `legacy_v1` | 早期单轮五维历史 | 保留原始总分、意见和档位，不重算。 |
| `two_round_v2` | 两轮改造期及第一轮 assignment | 第一轮按游戏性 60 + 创新性 40；已存在第二轮 V2 记录仍按其原三维规则显示。 |
| `two_round_v3` | 新建第二轮 assignment | 第二轮按五维综合复评 30/20/20/15/15。 |

关键表和概念：

| 对象 | 含义 |
| --- | --- |
| `project_pool` | 项目主档，一个真实项目一条。 |
| `project_materials` | 项目资料检查项。 |
| `project_status_history` | 项目状态、资料与人工调整历史。 |
| `meetings` | 评审会。 |
| `projects` | 某项目在某场评审会中的一次 assignment，包含轮次、尝试次数和评分版本。 |
| `scores` | 分项分数、问题、行动、Walker 加分和结论。 |
| `meeting_reviewers` | 会议评委快照，用于权限和完成度统计。 |
| `report_snapshots` | 报告快照及版本。 |

## 9. 部署与验收

- 正式地址：`https://scoresys.vercel.app`。
- 预览部署由 Vercel Deployment Protection 保护；对外验收必须使用该部署生成的 Shareable Link，不应直接发送普通预览域名。
- Shareable Link 仍进入系统自身登录页，只绕过 Vercel 外层验证。Hobby 方案同时仅允许一个有效公开链接，生成新链接会撤销旧链接。
- Vercel 的跨境网络和服务端区域可能带来中国网络访问延迟；功能验收应区分外层网络延迟与系统保存、汇总正确性。

## 10. 验收清单

1. 可创建、编辑、批量管理、归档和恢复项目，资料与状态刷新后保持真实值。
2. 可创建和编辑评审会，选择或快速创建项目入会，最多 12 个槽位并可拖拽排序。
3. 第一轮按 60/40 计算；第二轮新项目按五维 30/20/20/15/15 计算，风险预估必须参与统计。
4. 所有评委均能保存分数、问题和行动；Walker 能保存加分与结论；后台汇总实时更新。
5. Walker 结论不会提前隐藏已编排项目，截止日前评委仍可修改。
6. 报告中排名、表头、维度、评委参与和完成度与实时汇总一致，并可生成可追溯快照。
7. 旧项目与历史评分可读取，且已存在的 V2 第二轮记录不会被 V3 五维规则改写。
