# ScoreSys 项目池与跨评审会工作流设计

版本：v1.0

日期：2026-07-14

状态：业务设计已确认，待实施计划

关联文档：`立项评分系统2.0-PRD.md`

## 1. 目标

将 ScoreSys 从“以单次评审会为中心的打分工具”调整为“以项目为中心、评审会承载单轮评审”的立项评审系统，同时无损恢复旧版历史数据的展示。

本次设计实现以下结果：

1. 管理员先在待评审项目池建立项目并检查资料。
2. 资料检查通过后，管理员把项目安排到某场评审会，并明确本次评审轮次和尝试次数。
3. 每次评审会只保存本次轮次的评分、问题、建议和结论。
4. 同一项目可先后进入不同评审会，完成第一轮、第一轮重评、第二轮或第二轮重评。
5. 项目主档持续汇总跨会议历史、当前状态和最终结果。
6. 旧会议、旧项目和旧评分保持原样，可按旧版规则正确查看。
7. 新后台通过功能开关启用；试用不合适时可退回旧后台，不删除新旧业务数据。

## 2. 已确认业务规则

### 2.1 两轮评审

| 轮次 | 维度 | 维度满分 | 轮次满分 |
| --- | --- | ---: | ---: |
| 第一轮 | 游戏性 | 60 | 100 |
| 第一轮 | 创新性 | 40 | 100 |
| 第二轮 | 项目规划 | 40 | 100 |
| 第二轮 | 技术&美术 | 30 | 100 |
| 第二轮 | 风险预估 | 30 | 100 |

- 五位评委参与当前轮次的全部维度。
- 普通分项仍为 0-10 分。
- 游戏性中的“耐玩性”改为“可复玩性”，“可玩深度”改为“年龄适配度”。
- 新版创新性直接选择按 40 分满分缩放后的档位：10/16/24/30/40。
- 第一轮和第二轮独立计算、独立排序，不合并为 200 分或综合总分。
- Walker 在每轮给出“通过/重评/驳回”。
- 每轮最多一次重评机会；第一次结论为重评后，下一场会议使用同一轮次、尝试次数 2。尝试次数 2 不再提供“重评”，只能通过或驳回。
- 加分项继续存在，只作用于当前评审记录，不跨会议或跨轮累计。维度基础分满分为 100，加分单独展示，最终展示总分沿用当前封顶 100 的规则。

### 2.2 资料清单

| 资料项 | 要求 |
| --- | --- |
| 项目基础信息 | 必填 |
| 项目定位 | 必填 |
| 产品/玩法方案 | 必填 |
| 风险说明 | 必填 |
| 初步计划 | 必填 |
| 商业模式规划 | 选填 |
| 资源需求 | 选填 |
| 竞品或参考 | 选填 |

所有必填资料均标记为“已通过”后，项目才能进入正式评审会。

## 3. 方案选择

### 3.1 采用方案

新增独立的项目主档表 `project_pool`，保留现有 `projects` 表作为“某个项目在某场评审会中的评审记录”。两者通过 `projects.pool_project_id` 关联。

现有 `scores` 继续关联 `projects.id`，因此评分永远属于具体会议、具体轮次和具体尝试，不会因为项目主档归并而串分。

### 3.2 未采用方案

1. 仅在现有项目表增加 `root_project_id`：项目主档和会议记录仍混在一起，资料检查、状态追踪和跨会议历史难以维护。
2. 每条历史项目独立建档：最保守，但同一项目无法形成可靠的两轮链路，需要大量人工合并。
3. 将结果复制到独立结果表：会产生多个事实来源。结果池应由项目主档和最新有效结论派生，不复制评分结果。

## 4. 核心数据模型

### 4.1 `project_pool`：项目主档

项目主档是项目身份和流程状态的唯一来源。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | UUID | 项目唯一标识 |
| `name` | TEXT | 当前项目名称 |
| `submitter` | TEXT | 当前提报人 |
| `description` | TEXT | 当前项目说明 |
| `normalized_name` | TEXT | 归并匹配使用的标准化名称 |
| `normalized_submitter` | TEXT | 归并匹配使用的标准化提报人 |
| `match_key` | TEXT | `normalized_name + normalized_submitter` 的稳定匹配键 |
| `status` | TEXT | 当前项目流程状态 |
| `current_round` | SMALLINT/NULL | 未开始为空；评审中为 1 或 2 |
| `current_attempt` | SMALLINT | 当前轮次尝试次数，取值 1 或 2 |
| `latest_verdict` | TEXT/NULL | `approved/recheck/rejected` |
| `material_status` | TEXT | `unchecked/incomplete/complete/returned` |
| `material_note` | TEXT | 资料检查说明 |
| `created_at` | TIMESTAMPTZ | 创建时间 |
| `updated_at` | TIMESTAMPTZ | 最近更新时间 |
| `archived_at` | TIMESTAMPTZ/NULL | 归档时间，不做物理删除 |

`match_key` 用于迁移建议和重复提醒，不作为项目的永久业务主键，也不设置唯一约束。管理员可在确有需要时保留同名同提报人的两个独立项目。

### 4.2 `projects`：评审会项目记录

保留现有字段和现有主键，新增以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `pool_project_id` | UUID/NULL | 关联项目主档；历史空槽位为空 |
| `round_no` | SMALLINT/NULL | 新版为 1 或 2；旧版单轮历史为空 |
| `attempt_no` | SMALLINT | 新版为 1 或 2，默认 1 |
| `scoring_version` | TEXT | `legacy_v1` 或 `two_round_v2` |
| `assignment_status` | TEXT | `scheduled/scoring/awaiting_verdict/completed` |
| `migration_batch_id` | UUID/NULL | 历史迁移批次，用于审计和回滚 |

原有 `name`、`submitter` 和 `description` 作为当次评审快照保留。项目主档后续改名不会改写历史会议记录。

约束：

- 同一项目在同一评审会只能出现一次。
- 新版记录的 `round_no` 只能为 1 或 2。
- 新版记录的 `attempt_no` 只能为 1 或 2。
- 同一项目、同一轮次、同一尝试次数只能有一条评审记录。
- 每场评审会最多安排 12 个有效项目；空容量只在界面显示，不再写入空白项目记录。

### 4.3 `project_materials`：资料检查项

每个项目按资料项保存一条状态记录。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `project_id` | UUID | 关联项目主档 |
| `item_key` | TEXT | 稳定资料项键 |
| `required` | BOOLEAN | 是否必填 |
| `status` | TEXT | `missing/submitted/approved/needs_revision` |
| `note` | TEXT | 缺失或修改说明 |
| `checked_by` | TEXT/NULL | 最近检查人 |
| `checked_at` | TIMESTAMPTZ/NULL | 最近检查时间 |
| `updated_at` | TIMESTAMPTZ | 更新时间 |

主键为 `(project_id, item_key)`。项目整体 `material_status=complete` 必须由五个必填项全部为 `approved` 派生。

### 4.4 `project_status_history`：状态时间线

状态记录只追加，不覆盖和删除。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | UUID | 记录标识 |
| `project_id` | UUID | 项目主档 |
| `meeting_project_id` | UUID/NULL | 关联具体评审记录 |
| `meeting_id` | UUID/NULL | 关联评审会 |
| `event_type` | TEXT | 资料检查、安排入会、结论、人工调整等 |
| `from_status` | TEXT/NULL | 原状态 |
| `to_status` | TEXT | 新状态 |
| `operator_code` | TEXT | 操作人账号 |
| `note` | TEXT | 原因或说明 |
| `created_at` | TIMESTAMPTZ | 发生时间 |

管理员人工调整轮次、状态或结论时必须填写或确认说明，并写入时间线。已结束会议的原始评分不直接改写；必要修正以追加的更正记录体现。

### 4.5 `meeting_reviewers`：评委快照

创建评审会时，将当时参与的五位非管理员评委写入会议评委快照。完成度和报告使用快照计算，避免以后修改评委账号导致历史完成度变化。

### 4.6 迁移审计

新增 `project_migration_batches` 和 `project_migration_map`：

- 批次表记录迁移时间、执行人、预演统计、执行统计和状态。
- 映射表记录每个旧 `projects.id` 对应的 `project_pool.id` 和匹配键。
- 同一个旧项目只能存在一条有效映射。
- 回滚只处理指定批次，不扫描或猜测其他数据。

## 5. 项目状态机

项目主档使用以下主要状态：

```text
draft
  -> materials_pending
  -> ready_r1
  -> scheduled_r1
  -> r1_scoring
  -> ready_r2          第一轮通过
  -> r1_recheck_ready  第一轮首次重评
  -> rejected          第一轮驳回或第二轮驳回

ready_r2
  -> scheduled_r2
  -> r2_scoring
  -> initiation        第二轮通过
  -> r2_recheck_ready  第二轮首次重评
  -> rejected          第二轮驳回
```

状态规则：

1. `material_status` 未完成时不能进入 `ready_r1` 或安排评审会。
2. `ready_r1` 只能安排第一轮尝试 1。
3. `r1_recheck_ready` 只能安排第一轮尝试 2。
4. 第一轮通过后进入 `ready_r2`，由管理员手动安排新的第二轮评审会。
5. `ready_r2` 只能安排第二轮尝试 1。
6. `r2_recheck_ready` 只能安排第二轮尝试 2。
7. 尝试 2 的 Walker 结论不提供重评选项。
8. 第二轮通过后进入 `initiation`，代表进入立项流程。
9. 管理员可人工调整状态，但必须记录操作人、时间、原状态、新状态和说明。

## 6. 评分版本与历史兼容

### 6.1 `legacy_v1`

历史单轮评审保持原规则：

| 维度 | 满分 |
| --- | ---: |
| 可玩性 | 30 |
| 创新性 | 20 |
| 项目规划 | 20 |
| 技术&美术 | 15 |
| 风险评估 | 15 |

旧创新性档位保持 4/6/10/14/20。旧名称“风险性”“风险评估”和“可玩性”通过旧版别名解析，不改写数据库字段。

历史计算规则：

1. 同一评委、同一维度存在分项评分时，优先按分项计算。
2. 没有分项评分但存在旧版维度总分时，使用维度总分。
3. 分项和维度总分不能重复累加。
4. 旧版 `__problems__`、`__actions__`、`__verdict__` 在历史详情中原样显示。
5. 无法可靠推断应评人数的旧会议显示“历史记录数”，不生成误导性的完成百分比。
6. 历史单轮记录的 `round_no` 保持为空，不强行归入第一轮或第二轮，也不参与新版项目状态自动流转。

### 6.2 `two_round_v2`

- 普通评分键使用 `r1::` 或 `r2::` 前缀。
- 特殊字段按具体评审记录和轮次保存。
- 汇总接口根据 `projects.scoring_version` 选择规则集，不能用当前规则重新解释旧评分。
- 报告同时支持旧版历史卡片和新版分轮报告，但旧版数据不与新版轮次排名混排。

## 7. 历史迁移设计

### 7.1 已完成的生产数据核验

恢复和预演时确认：

- 8 场会议，均已恢复为可见状态。
- 91 条项目记录，其中 42 条为已填写项目。
- 800 条评分记录，无孤立评分。
- 797 条旧格式评分，3 条新版轮次评分。
- 按“项目名称 + 提报人”标准化匹配后形成 35 个项目主档。
- 7 组项目发生归并。
- 0 组同名但提报人不同的冲突。
- 0 组同一场会议内的错误重复归并。

### 7.2 匹配规则

名称和提报人分别执行：Unicode NFKC 标准化、首尾空格清理、连续空白合并、大小写统一。随后组合为 `match_key`。

不采用拼音、模糊字符串、别名或语义相似度归并。没有精确命中时创建独立项目主档，并在后台提供重复提醒和人工合并入口。

### 7.3 主档字段选择

- 项目名称和提报人采用标准化匹配组中最近一条非空记录的显示值。
- 项目说明采用最近一条非空说明。
- 每场会议中的原名称、提报人、说明继续保存在原 `projects` 行中作为快照。
- 历史状态不根据旧分数猜测；未存在明确结论时保留为空或“历史数据”。

### 7.4 执行步骤

1. 保留当前恢复前快照和 SHA256 校验文件。
2. 执行仅新增表、字段、索引和约束的数据库迁移。
3. 部署兼容旧版和新版的双读代码，项目池功能开关保持关闭。
4. 再次运行只读预演，要求统计结果与本设计一致。
5. 在单个数据库事务中创建迁移批次、35 个项目主档、42 条映射并更新关联字段。
6. 验证会议数、项目数、评分数、孤立评分数和映射唯一性。
7. 开启项目池功能开关，进行管理员、评委、Walker 和报告全流程检查。

迁移脚本必须幂等：同一个成功批次不能重复生成项目主档或重复更新映射。

### 7.5 回滚

首选回滚仅关闭项目池功能开关并重新部署旧后台视图。由于新版评审仍写入现有 `projects` 和 `scores`，关闭功能开关不会删除试用期间产生的评审数据。

如需回滚数据库关联：

1. 指定迁移批次。
2. 将该批次关联的旧 `projects.pool_project_id` 和 `migration_batch_id` 置空。
3. 删除该批次映射和未被试用期新记录引用的项目主档、资料项与状态记录。
4. 不删除会议、原项目行或评分。
5. 回滚后重新验证 8 场会议、91 条原项目记录、800 条原评分和 0 条孤立评分。

如果项目池启用后已经产生新项目或新评审，默认只关闭功能开关，不执行数据库删除。

## 8. 后台信息架构

### 8.1 待评审项目池管理

- 新建和编辑项目主档。
- 展示资料完整度、当前状态、当前轮次和最近更新时间。
- 按资料状态和流程状态筛选。
- 在项目详情中逐项检查八类资料。
- 必填资料未通过时，“安排评审会”不可用，并显示缺失项。

### 8.2 评审会管理

- 创建、编辑、切换和归档评审会。
- 显示“已安排 X/12”，不在数据库创建空槽位。
- 从符合条件的项目池中选择项目。
- 每个项目明确选择第一轮或第二轮；尝试次数由系统根据项目历史自动确定，不允许管理员随意跳过。
- 入会前显示项目最近结论和上轮问题/建议摘要。
- 已开始评分的评审记录不能从会议中直接移除。

### 8.3 结论与报告

- 以评审会为范围展示本次项目排序、完成度、Walker 结论、问题和建议。
- 第一轮和第二轮分别使用 100 分制，不混排。
- Walker 结论即时更新项目主档和状态时间线。
- 管理员手工更改结论需要额外确认，并写入更正记录。
- 报告读取当前数据库状态，保留本次评审快照和生成时间。

### 8.4 已评审项目池管理

- 以项目主档为一行，不按会议重复展示。
- 项目详情按时间展示每场评审会、轮次、尝试次数、分数、结论、问题、建议和状态变化。
- 评委原始问题和建议保持只读。
- 管理员整理后的问题和建议按轮次保存，用于后续安排和报告。
- 历史 `legacy_v1` 记录使用独立标签展示。

### 8.5 结果池

结果池是项目池的派生视图，包含三个标签：

- 通过项目：最近有效结论为通过；可按第一轮通过待安排第二轮、第二轮通过进入立项流程继续筛选。
- 待重评项目：最近有效结论为重评，并显示必须安排的轮次和尝试次数 2。
- 驳回项目：最近有效结论为驳回。

结果池不复制评分或项目数据。

## 9. 评委与 Walker 页面

- 评委先选择当前评审会，只看到该会议已安排的有效项目。
- 每个项目的轮次由评审记录决定，不从项目主档临时推断。
- 第一轮只显示游戏性和创新性；第二轮只显示项目规划、技术&美术和风险预估。
- 五位会议评委都能填写当前轮次全部维度、存在问题和修改建议。
- Walker 页面底部显示当前轮次的加分项和结论。
- 尝试次数 2 时隐藏“重评”按钮。
- 已锁定的历史评审记录只读。

## 10. API 边界

新增或调整以下服务端能力：

| API | 职责 |
| --- | --- |
| `/api/project-pool` | 项目池查询、新建和编辑 |
| `/api/project-pool/[id]/materials` | 资料清单读取和检查 |
| `/api/project-pool/[id]/history` | 跨会议评审与状态时间线 |
| `/api/meeting-assignments` | 安排项目入会、移除未开始项目 |
| `/api/results` | 通过、待重评、驳回派生列表 |
| `/api/summary` | 按评分版本和会议生成汇总 |
| `/api/scores` | 按具体评审记录保存评分和结论 |

安排入会和 Walker 结论流转必须在服务端校验状态，不能只依赖页面按钮禁用。

安排入会使用数据库事务或受约束的数据库函数，原子检查以下条件：资料完整、状态允许、轮次正确、尝试次数正确、会议容量未超过 12、同项目未重复入会。

## 11. 错误处理与数据保护

- 所有写接口返回明确的业务错误，例如“必填资料未通过”“第一轮尚未通过”“本轮重评机会已使用”“会议已满”。
- API 保存失败时不提前更新项目主档状态。
- 评分保存继续使用唯一键 upsert，避免重复请求产生重复分数。
- 状态流转和结论保存必须在同一事务中完成；任一步失败则整体回滚。
- 项目主档采用归档而非物理删除。存在评审历史的项目不能物理删除。
- 会议和项目的高风险删除入口保留二次确认；新版不提供自动到期物理清理。
- 生产迁移前后都生成记录数和孤立关联检查报告。

## 12. 功能开关与发布

服务端使用 `PROJECT_POOL_V2_ENABLED` 控制新版后台入口和新版 API 写能力：

- `false`：旧后台继续工作，新表和新增字段保持不使用。
- `true`：启用项目池后台、跨会议项目历史和新版安排流程。

评分兼容读取不受功能开关影响；旧评分恢复展示应始终生效。

发布顺序：数据库新增结构 -> 双读代码 -> 历史兼容验证 -> 项目池迁移 -> 开启功能开关。不得在同一次未经验证的发布中同时执行破坏性结构变更和界面切换。

## 13. 测试策略

### 13.1 单元测试

- `legacy_v1` 五个维度正确计算为 100 分。
- 旧版分项和维度总分不会重复累加。
- `two_round_v2` 第一轮和第二轮分别正确计算为 100 分。
- 创新性旧版档位与新版档位按各自规则解析。
- 项目名称和提报人标准化匹配稳定、无模糊归并。
- 轮次、尝试次数和结论状态机覆盖所有允许与禁止路径。

### 13.2 API 与集成测试

- 必填资料不完整的项目不能入会。
- 第一轮未通过的项目不能安排第二轮。
- 每轮首次重评可创建尝试 2，尝试 2 不能再次重评。
- 同一项目不能重复加入同一会议。
- 第 13 个项目不能加入同一会议。
- Walker 结论同时更新评审记录、项目主档和状态历史。
- 管理员更正结论需要确认并留下记录。
- 五位评委的数据都能即时出现在会议汇总和报告中。

### 13.3 迁移与回归测试

- 预演得到 42 条旧项目映射、35 个主档和 7 组归并。
- 重复执行迁移不会增加主档或映射。
- 迁移前后评分记录数相同，孤立评分数为 0。
- 797 条旧格式评分在旧会议详情中可见并按旧规则显示。
- 关闭功能开关后旧后台仍能读取会议、项目和评分。
- 新版项目从资料检查到两轮通过的完整路径可在两场或更多会议中完成。

## 14. 验收标准

1. 管理员能在独立项目池中新建项目并完成资料检查。
2. 必填资料未通过的项目不能加入评审会。
3. 管理员能从项目池选择项目加入会议，并设置第一轮或第二轮。
4. 同一项目的两轮可以位于不同会议，且历史连续关联。
5. 每场会议最多安排 12 个项目，数据库不再创建空白占位项目。
6. 第一轮通过项目回到项目池，等待管理员安排第二轮。
7. 两轮各允许一次跨会议重评，第二次尝试不能继续重评。
8. 已评审项目池能按项目展示全部会议、分数、问题、建议和结论。
9. 结果池能正确派生通过、待重评和驳回项目。
10. 旧版会议和 797 条旧格式评分恢复可见，原始记录不被改写。
11. 迁移后会议、项目、评分数量符合校验报告，孤立评分为 0。
12. 关闭功能开关可退回旧后台，且试用期间数据不丢失。

- 全面重做登录和权限体系；本期沿用现有角色并强化新增写接口的服务端校验。
- 删除旧表、旧字段或旧评分键。
